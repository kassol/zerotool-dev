<h2>使用方法</h2>
<ol>
  <li>在搜索框输入 stack 名称——例如 <code>node</code>、<code>macos</code>、<code>jetbrains</code>。</li>
  <li>勾选所需模板，已选项以可移除 chip 形式显示在顶部。</li>
  <li>（可选）展开「追加自定义规则」补上项目特有路径。</li>
  <li>合并后的 <code>.gitignore</code> 在下方实时更新——点击 <strong>复制</strong> 或 <strong>下载</strong>。</li>
  <li>把文件放到仓库根目录并 commit。</li>
</ol>

<h2>什么是 .gitignore 文件？</h2>
<p>
  <code>.gitignore</code> 列出 Git 在 stage 时应忽略的路径模式——通常是构建产物、操作系统元数据、IDE 缓存、依赖目录与密钥文件。模式遵循 gitignore glob 语法（<code>*.log</code>、<code>build/</code>、<code>!keep-this.log</code>），按目录生效。每个 Git 仓库都应该提交一份，以保证 clone 可复现，避免泄漏环境噪声。
</p>

<h2>合并逻辑</h2>
<p>
  选中的模板按字母排序拼接。每段会输出 <code>### Name.gitignore</code> 头部，方便追溯规则来源。跨模板重复的规则行在首次出现后被丢弃，注释与空行原样保留。自定义规则在最后以 <code>### Custom additions</code> 段呈现。生成结果开头带一行指向 ZeroTool 与 github/gitignore 源的 attribution——不喜欢可以直接删掉。
</p>

<h2>为什么用预打包模板</h2>
<p>
  请求时去抓 github/gitignore 会带来网络延迟与速率限制风险。我们在构建期一次性快照仓库、去重 ID、打成单个 TypeScript 模块——可预测、可审计、零运行时依赖。刷新时手动跑同步脚本，从上游拉新快照后一次 commit 即可。
</p>
