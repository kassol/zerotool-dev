<h2>How to Use</h2>
<ol>
  <li>Enter a domain — bare hostname, full URL, or trailing-dot form all work; the tool normalises automatically.</li>
  <li>Pick a record type. <strong>ALL</strong> fans out parallel queries for A, AAAA, CNAME, MX, TXT, NS, SOA, and CAA in one click.</li>
  <li>Pick a resolver: <strong>Cloudflare (1.1.1.1)</strong> or <strong>Google (8.8.8.8)</strong>.</li>
  <li>Click <strong>Lookup</strong> or press <kbd>Enter</kbd>.</li>
  <li>Switch between <strong>Records</strong>, <strong>Raw</strong> (dig-style output), and <strong>JSON</strong> (raw DoH response) tabs.</li>
</ol>

<h2>Record types supported</h2>
<ul>
  <li><strong>A</strong> — IPv4 address.</li>
  <li><strong>AAAA</strong> — IPv6 address.</li>
  <li><strong>CNAME</strong> — canonical alias pointing at another name.</li>
  <li><strong>MX</strong> — mail exchanger with priority.</li>
  <li><strong>TXT</strong> — arbitrary text. Hosts SPF, DKIM, DMARC, domain verification, and HTTP/3 origin signals.</li>
  <li><strong>NS</strong> — authoritative name servers for the zone.</li>
  <li><strong>SOA</strong> — Start of Authority: primary NS, contact, serial, refresh timers.</li>
  <li><strong>CAA</strong> — which Certificate Authorities may issue for the domain.</li>
  <li><strong>PTR</strong> — reverse DNS. Query the <code>x.y.z.w.in-addr.arpa</code> name yourself; the tool issues a plain PTR query.</li>
  <li><strong>SRV</strong> — service location with priority, weight, port, and target.</li>
</ul>

<h2>Why DNS-over-HTTPS</h2>
<p>
  Browsers cannot open raw UDP/53 sockets, which is why no in-browser tool can speak classic DNS.
  DoH wraps DNS messages in HTTPS, so a regular <code>fetch()</code> call can ask a public resolver
  for any record. Cloudflare and Google both expose a JSON profile (<code>application/dns-json</code>
  for Cloudflare, native JSON for Google) with permissive CORS — this tool does nothing more than
  call those endpoints and render the response.
</p>

<h2>Reading the summary pills</h2>
<ul>
  <li><strong>NOERROR / NXDOMAIN / SERVFAIL</strong> — the DNS RCODE. NXDOMAIN is a valid answer meaning "domain does not exist"; SERVFAIL means the resolver could not answer.</li>
  <li><strong>N records</strong> — how many answer records came back. Zero with NOERROR means the type exists for the zone elsewhere but not for the requested name.</li>
  <li><strong>DNSSEC ✓</strong> — the AD flag is set. The resolver validated signatures.</li>
  <li><strong>Resolver</strong> — which DoH endpoint served the response.</li>
  <li><strong>Latency</strong> — round-trip time from <code>fetch()</code> start to JSON parse.</li>
</ul>

<h2>Typical use cases</h2>
<ul>
  <li>Verifying that an A or CNAME change has propagated past a specific resolver.</li>
  <li>Confirming MX, SPF (TXT), DKIM (TXT), and DMARC (TXT) before launching a new sending domain.</li>
  <li>Comparing the records two public resolvers see — Cloudflare vs Google can disagree during a propagation window.</li>
  <li>Checking CAA before issuing a certificate to avoid the "CAA mismatch" failure mode.</li>
  <li>Sanity-checking NS delegation when moving zones between registrars.</li>
</ul>

<h2>Limits by design</h2>
<p>
  This is a single-shot resolver client. It does not perform automatic ARPA conversion for reverse
  IP lookups (input the in-addr.arpa name yourself for PTR), it does not query authoritative
  servers directly (it asks a public recursive), and it does not run history, WHOIS, blocklist, or
  propagation comparisons across many resolvers. For propagation maps use
  <a href="https://www.whatsmydns.net/">whatsmydns.net</a>; for authoritative-only lookups use
  <code>dig @ns1.example.com</code> locally.
</p>
