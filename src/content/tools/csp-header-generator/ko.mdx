<h2>사용 방법</h2>
<ol>
  <li><strong>프리셋</strong>을 선택합니다. Strict는 OWASP의 "strict CSP" 권장 구성, Empty는 빈 정책에서 시작합니다.</li>
  <li><strong>모드</strong>를 전환합니다. Report-Only로 시작해 위반은 보고하되 실제 사용자 흐름은 막지 않도록 합니다.</li>
  <li>각 지시문에서 키워드 칩(<code>'self'</code>, <code>'strict-dynamic'</code>, <code>'none'</code>) 또는 스킴 칩(<code>https:</code>, <code>data:</code>)을 클릭하거나, 호스트/경로를 입력하고 <kbd>Enter</kbd>를 누르세요.</li>
  <li><strong>해시 계산기</strong>로 특정 인라인 스크립트/스타일을 허용하는 해시를 만들고, <code>+ nonce</code> 버튼으로 새 nonce를 즉시 추가할 수 있습니다.</li>
  <li>출력 탭을 전환해 필요한 형식을 복사하세요: <strong>HTTP 헤더</strong>, <strong>HTML &lt;meta&gt;</strong>, <strong>Express (helmet)</strong>, <strong>Nginx</strong> <code>add_header</code>.</li>
</ol>

<h2>Strict CSP 시작 구성</h2>
<p>Strict 프리셋은 세 가지 규칙으로 짜여 있습니다: <code>script-src</code>에 <code>'self' 'strict-dynamic'</code>, <code>object-src</code>에 <code>'none'</code>, <code>base-uri</code>에 <code>'self'</code>. 요청별 nonce 또는 스크립트별 해시와 결합하면 대부분의 반사형/저장형 XSS 페이로드를 차단할 수 있습니다. 피싱 형 공격까지 막고 싶다면 <code>form-action</code>은 <code>'self'</code>, <code>frame-ancestors</code>는 <code>'none'</code>으로 잠그세요.</p>

<h2>흔한 함정</h2>
<ul>
  <li><strong><code>'none'</code>은 단독으로만 작동합니다.</strong> 다른 소스와 함께 적으면 브라우저가 해당 지시문을 무시합니다.</li>
  <li><strong><code>frame-ancestors</code>는 HTTP 헤더 전용.</strong> <code>&lt;meta&gt;</code>로 보내면 무시되며, <code>report-uri</code>, <code>report-to</code>, <code>sandbox</code>도 동일합니다.</li>
  <li><strong>해시/nonce가 있으면 <code>'unsafe-inline'</code>은 무시됩니다.</strong> 최신 브라우저는 더 안전한 쪽을 우선합니다.</li>
  <li><strong><code>default-src</code>는 반드시 정의하세요.</strong> 누락된 fetch 지시문은 <code>default-src</code>로 폴백하며, 이것이 없으면 새로 추가될 지시문에도 폴백이 적용되지 않습니다.</li>
  <li><strong>커스텀 호스트에 세미콜론을 넣지 마세요.</strong> 세미콜론은 정책을 끊어 이후 모든 지시문을 손상시킵니다.</li>
</ul>

<h2>해시 vs. Nonce</h2>
<p><strong>정적 인라인 콘텐츠</strong>(애널리틱스 스니펫, 크리티컬 CSS, 서버 렌더링되는 보조 스크립트)에는 해시를 사용합니다. 정적 HTML과 엣지 캐시와 잘 맞고, 요청별 값을 필요로 하지 않습니다.</p>
<p><strong>요청마다 새로 렌더링되는 콘텐츠</strong>에는 nonce를 사용합니다. 서버가 헤더에 <code>'nonce-XYZ'</code>를 넣고, 동일한 값을 모든 <code>&lt;script nonce="XYZ"&gt;</code>에 새깁니다. <code>'strict-dynamic'</code>과 함께 사용하면 신뢰된 스크립트의 자식 스크립트도 신뢰가 상속돼 거대한 호스트 허용 목록을 유지할 필요가 없습니다.</p>

<h2>배포 치트시트</h2>
<ul>
  <li><strong>Nginx</strong>: Nginx 탭 내용을 <code>server</code> 블록에 붙여넣고 reload. <code>always</code> 플래그는 오류 응답에도 헤더를 강제합니다.</li>
  <li><strong>Apache</strong>: <code>.htaccess</code>나 <code>httpd.conf</code>에 <code>Header always set Content-Security-Policy "..."</code>를 작성합니다.</li>
  <li><strong>Express / Node</strong>: helmet 스니펫을 붙여넣고, <code>useDefaults</code>를 <code>false</code>로 설정해 정책이 helmet 기본값과 조용히 병합되지 않도록 하세요.</li>
  <li><strong>Cloudflare Pages / Vercel / Netlify</strong>: HTTP 헤더 형식을 각 플랫폼의 헤더 설정(<code>_headers</code>, <code>vercel.json</code>, <code>netlify.toml</code>)에 붙여넣습니다.</li>
</ul>
