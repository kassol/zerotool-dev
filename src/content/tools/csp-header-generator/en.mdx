<h2>Strict CSP starter</h2>
<p>The Strict preset is built around three rules: <code>'self' 'strict-dynamic'</code> on <code>script-src</code>, <code>'none'</code> on <code>object-src</code>, and <code>'self'</code> on <code>base-uri</code>. Combined with per-request nonces or per-script hashes, this layout resists most reflected and stored XSS payloads. Add <code>'self'</code> to <code>form-action</code> and <code>'none'</code> to <code>frame-ancestors</code> to lock down phishing-style abuse.</p>

<h2>Common pitfalls</h2>
<ul>
  <li><strong><code>'none'</code> is exclusive.</strong> If you list any other source alongside <code>'none'</code>, the browser drops the directive.</li>
  <li><strong>frame-ancestors is HTTP only.</strong> Browsers ignore it when CSP is delivered via <code>&lt;meta&gt;</code>. The same applies to <code>report-uri</code>, <code>report-to</code>, and <code>sandbox</code>.</li>
  <li><strong>Hashes and nonces void <code>'unsafe-inline'</code>.</strong> Modern browsers fall back to the safer of the two when both are present.</li>
  <li><strong>Always set <code>default-src</code>.</strong> Missing fetch directives fall back to <code>default-src</code>; without it the browser applies no fallback for new directives added later.</li>
  <li><strong>Custom hosts cannot contain semicolons.</strong> A semicolon would terminate the policy and corrupt every directive after it.</li>
</ul>

<h2>Hashes vs. nonces</h2>
<p>Use <strong>hashes</strong> for static inline content that ships with the page — analytics snippets, critical CSS, server-rendered helpers. Hashes work for static HTML and edge-cached responses because they do not need any per-request value.</p>
<p>Use <strong>nonces</strong> for dynamic content rendered per request. The server emits a <code>Content-Security-Policy</code> header containing <code>'nonce-XYZ'</code> and stamps the same value on every <code>&lt;script nonce="XYZ"&gt;</code> it renders. Pair nonces with <code>'strict-dynamic'</code> so descendant scripts inherit trust without you maintaining a host allow-list.</p>

<h2>Deployment cheat sheet</h2>
<ul>
  <li><strong>Nginx</strong>: copy the Nginx tab into your <code>server</code> block and reload — the <code>always</code> flag forces the header on error responses too.</li>
  <li><strong>Apache</strong>: use <code>Header always set Content-Security-Policy "..."</code> in <code>.htaccess</code> or <code>httpd.conf</code>.</li>
  <li><strong>Express / Node</strong>: paste the helmet snippet, and remember to disable <code>useDefaults</code> so your policy is not silently merged with helmet defaults.</li>
  <li><strong>Cloudflare Pages / Vercel / Netlify</strong>: paste the HTTP header form into your platform's headers config (<code>_headers</code>, <code>vercel.json</code>, <code>netlify.toml</code>).</li>
</ul>
